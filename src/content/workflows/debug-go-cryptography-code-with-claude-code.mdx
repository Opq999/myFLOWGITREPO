## What you'll get
You will get a working Go cryptography implementation with a complex, low-level bug identified and fixed. This approach leverages AI's ability to quickly analyze large codebases and pinpoint subtle logical errors, especially in well-scoped problems with failing tests.

## Tools you need
*   **Claude Code** (freemium): An AI assistant for coding tasks, used here to debug Go code.
*   **Go** (free): The programming language used for the cryptography implementation.

## Steps
1.  **Prepare your Go project**: Ensure your Go project is set up and the failing tests are reproducible.
    *   What to do: Have your Go project directory open in your terminal or IDE. Make sure the `go test` command for your specific module is ready to run and consistently shows failures.
    *   What you should see: When you run `bin/go test crypto/internal/fips140/mldsa` (or your equivalent test command), you should see output indicating `FAIL` for your tests, similar to `FAIL: TestVector (0.00s) mldsa_test.go:47: Verify: mldsa: invalid signature`.
2.  **Open Claude Code**: Access the Claude Code interface.
    *   What to do: Go to `https://claude.ai` in your web browser. Log in or sign up if prompted. The author used Claude Code v2.0.28 with Opus 4.1, which is a specific model. If you have access to different models, choose the most capable one available to you.
    *   What you should see: The Claude chat interface, ready for you to enter a prompt.
3.  **Craft your debugging prompt**: Provide Claude with context about the bug and where to find the code.
    *   What to do: Copy the following prompt and paste it into Claude. Replace `[your Go module path]` with the actual path to your module (e.g., `crypto/internal/fips140/mldsa`) and `[your code directory]` with the path to your source code (e.g., `src/crypto/internal/fips140/mldsa`).

    ```
    I implemented ML-DSA in the Go standard library, and it all works except that verification always rejects the signatures. I know the signatures are right because they match the test vector. You can run the tests with bin/go test [your Go module path] You can find the code in [your code directory] Look for potential reasons the signatures don't verify. ultrathink I spot-checked and w1 is different from the signing one.
    ```

    *   What you should see: Claude will start processing your request.
4.  **Wait for Claude's analysis and fix**: Claude will analyze your code and identify the bug.
    *   What to do: Allow Claude a few minutes to process. The author noted that Claude "pinged me a few minutes later with a complete fix."
    *   What you should see: Claude should return an explanation of the bug and a proposed fix. In the author's case, Claude identified that `w1Encode` was being reused in `Verify` where `UseHint` already produced the high bits, effectively taking the high bits of `w1` twice.
5.  **Review and apply the fix**: Understand Claude's proposed solution and integrate it into your code.
    *   What to do: Carefully read Claude's explanation and the suggested code changes. The author refined Claude's "mediocre fix" by refactoring `w1Encode` to take high bits as input, which was clearer and more efficient. Apply the fix to your Go source code.
    *   What you should see: Your Go code editor will show the changes you've made based on Claude's suggestions.
6.  **Run tests to confirm the fix**: Verify that the bug is resolved.
    *   What to do: Go back to your terminal or IDE and run your Go tests again using the command from Step 1 (e.g., `bin/go test crypto/internal/fips140/mldsa`).
    *   What you should see: Your tests should now pass, indicating that the signature verification issue has been resolved. The output should no longer show `FAIL`.

## Notes & variations
*   **Free-tier alternatives**: While the author used a free Claude Max subscription, a free tier of Claude is generally available at `https://claude.ai`. Other AI models like ChatGPT (`https://chatgpt.com`) or Gemini (`https://gemini.google.com`) could also be tried, though their effectiveness with low-level cryptography might vary.
*   **Common mistake**: Not providing enough context or the actual code. AI models need the full code snippets or clear instructions on where to find them (if they have web browsing capabilities) to effectively debug. Simply describing the problem without the code will lead to generic, unhelpful suggestions.
*   **Tip for better results**: For complex bugs, break down the problem into smaller, more manageable parts if the AI struggles with the whole. Also, explicitly tell the AI what you've already tried or what you suspect, as the author did with "ultrathink I spot-checked and w1 is different from the signing one." This guides the AI's focus.
